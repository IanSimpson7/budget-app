<interfaces>
Contracts this plan CREATES. Downstream plans (02/03/04/05) import these.
food.types.ts holds pure types, NO Dexie imports, mirroring src/storage/schema.ts Readonly + discriminated-union style.

MealType = 'decomposed' | 'flat-cost'
IngredientTag = 'macro-bearing' | 'flavor-condiment'

MealDefinition (Readonly):
  id?: number
  mealName: string        // NORMALIZED lowercase + trimmed — the SMC-join key (Pitfall 5)
  type: MealType
  ingredients: string[]   // macro-bearing ingredient names; [] for flat-cost meals
  flatCost?: number       // required when type==='flat-cost'; unset triggers fallback-high (D-04)

UnitCostEntry (Readonly):
  ingredientName: string  // key space shared with PortionEntry + MealDefinition.ingredients[]
  costPerUnit: number     // dollars/unit; 0 is treated as unpriced and triggers fallback-high (FOOD-08)
  unit: string            // 'lb' | 'oz' | 'each' | ...
  tag: IngredientTag      // flavor-condiment entries are EXCLUDED from per-meal sum (D-05)
                          // C1 DEFAULT (I-05): any newly-created entry MUST default to 'macro-bearing'
                          // (included in the floor cost). Ian opts INTO 'flavor-condiment'.
                          // Defaulting to flavor would silently exclude the ingredient from the
                          // sum = undercount = C1 risk. Document this default in food.types.ts.

PortionEntry (Readonly):
  ingredientName: string
  portionSize: number     // expressed in UnitCostEntry.unit

FoodFloorMeta (Readonly):  // C1: NO writable floor field — engine-written metadata only
  lastComputedFloor: number
  allTimeHighWater: number
  lastRefinedFromReceipts: string | null   // ISO datetime or null

FlavorLine (Readonly):
  amount: number          // monthly PROTECTED flavor/condiment budget (seed ~50)

Normalization contract (document in food.types.ts): normalizeMealName(s) = s.trim().toLowerCase().
The 14 seed meals are stored normalized; the parser normalizes SMC names the same way before lookup.
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input to storage | Unit-cost / portion / flavor numbers typed in the config UI cross into persisted state |
| JSON import to storage | A user-supplied backup file is read and replaces all state |
| cost engine to foodFloorMeta | Engine-written metadata is the ONLY writer of floor-related persisted values |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | food floor value | mitigate | No setFoodFloor/decreaseFoodFloor exists on storage.ts; FoodFloorMeta has no writable floor field, only engine-written lastComputedFloor/allTimeHighWater (V6 absence-proof test). C1. |
| T-04-02 | Tampering | numeric config inputs | mitigate | Number.isFinite guard before persist on unit-cost/portion/flavor; rejects NaN/Infinity that could zero or corrupt the floor sum. |
| T-04-03 | Denial of Service | JSON import | mitigate | replaceAll runs in a single Dexie transaction (atomic); a malformed mealDefinitions array is guarded by Array.isArray before restore (no partial-state corruption). |
| T-04-04 | Information Disclosure | local IndexedDB | accept | Single-user local-only app, no credentials (C2), no network, no exfiltration channel. |
</threat_model>
