## C1 Visual Contract ‚Äî Non-Negotiable

The food floor is governed by an inviolable clinical constraint (C1). The following visual rules enforce C1 structurally ‚Äî they are not stylistic preferences:

1. **The protected food floor dollar value is rendered read-only at all times.** No `<input>`, no `contenteditable`, no adjacent pencil/edit icon on the floor line. The value is a `<span>` or equivalent non-interactive element.
2. **A lock icon (`lucide-react` `Lock` icon, 16px) appears left of the floor value.** It is decorative (aria-hidden), but its presence must never be conditional ‚Äî it must always render when the floor value renders.
3. **The explainer copy "Computed from your meal plan ‚Äî protected" appears directly below or beside the floor value** in `text-xs text-text-secondary`. It must not be hidden behind a toggle or collapsed by default.
4. **No UI anywhere in Phase 4 presents a "cut", "reduce", "trim", or "save on food" affordance.** This includes: button labels, helper text, empty-state suggestions, and error recovery copy.
5. **The config surface (unit-cost map, portion model, meal-definition table) is framed as accuracy/convergence tooling**, never as a budget-lever. Its heading must read "Meal cost configuration" (or equivalent), not "Adjust food budget" or "Food spending controls".
6. **The flavor line ($50/mo) is labeled "Flavor & condiments ‚Äî protected"** and must have an affordance only for editing its monthly dollar amount upward, never downward framing or gating affordance. It renders in the PROTECTED section alongside the locked floor.

Executor checklist: UI-SPEC checker will verify each of these 6 conditions independently.

## Surfaces

Phase 4 introduces two new surfaces. Both slot into the existing HashRouter and AppShell.

### Surface 1: Food Panel (`/food` or embedded in `/dashboard`)

**Routing decision (Claude's discretion):** Add a `Food` nav link to AppShell pointing to `/food`. This keeps the food floor always accessible without overloading the dashboard pre-Phase 5 integration. The dashboard's "Survival floor" MetricCard (already present) will update automatically when `foodFloorAtom` replaces `floors.foodSeed` ‚Äî no dashboard layout change required in Phase 4.

**Layout:** Single-column `flex flex-col gap-sp-6`. Max-width inherited from AppShell (`max-w-[960px]`).

**Sections, top to bottom:**

1. **Page heading** ‚Äî `font-display text-[20px]`: "Food ¬∑ {Month Year}"
2. **Protected floor card** ‚Äî `bg-surface-raised border border-surface-border rounded-sm p-sp-3`. Contains:
   - Label row: `Lock` icon (16px, `text-text-secondary`, `aria-hidden="true"`) + "Protected floor" in `font-sans text-xs font-semibold text-text-secondary`
   - Value: `font-mono text-[20px] leading-[1.2] text-text-primary` ‚Äî the computed floor dollar amount
   - Explainer: `font-sans text-xs text-text-secondary` ‚Äî "Computed from your meal plan ‚Äî protected"
   - Status badge (see Badge spec below) ‚Äî inline, right-aligned on the same row as the label, or on its own row on narrow screens
3. **Flavor line row** ‚Äî inside the same protected card or as a sibling card: "Flavor & condiments ‚Äî protected" + `font-mono text-[20px]` amount + `<input>` for editing the monthly amount (upward framing only ‚Äî no "reduce" label)
4. **Gateable food layer row** ‚Äî `bg-surface-raised border border-surface-border rounded-sm p-sp-3`: Label "Discretionary food (gateable)" + summed dollar amount from Phase 3 gateable food expense lines. Read-only display ‚Äî no edit affordance here (editing belongs in `/expenses`). A `text-xs text-text-secondary` subtext "From your expense lines ‚Äî edit in Expenses" with a link to `/expenses`.
5. **Empty state** (when no meal plan detected): see Copywriting Contract.

**Status Badge (D-11):**

| State | Badge classes | Label |
|-------|--------------|-------|
| Clean (all resolved) | `bg-success bg-opacity-10 border border-success text-success font-sans text-xs rounded-sm px-sp-2 py-sp-1` | "Plan current" |
| Needs attention | `bg-warning bg-opacity-10 border border-warning text-warning font-sans text-xs rounded-sm px-sp-2 py-sp-1` | "Needs attention" |

The badge is a `<button>` (or disclosure trigger) that expands an itemized detail list. When expanded, the list renders each gap as a separate row:
- Unpriced ingredient: "Ingredient '{name}' has no unit cost ‚Äî add it in Meal config"
- Undefined meal: "Meal '{name}' has no definition ‚Äî add ingredients in Meal config"
- Stale plan: "No meal plan covers today ‚Äî showing fallback floor (last known: {date})"

Each row uses `font-sans text-sm text-warning`. The list is visually contained in a `bg-surface-raised border border-warning rounded-sm p-sp-3 mt-sp-2` panel. An expanded state badge becomes "Plan current (tap to collapse)" / "Needs attention (tap to collapse)".

The badge toggle button has `min-h-[44px]` and `aria-expanded` on the control element. The detail list has `role="list"`.

### Surface 2: Food Config (`/food/config`)

A sub-route of `/food`. Accessible via a "Meal cost configuration" link/button on the food panel. The AppShell nav does NOT add a direct config link ‚Äî config is reached through the food panel only.

**Page heading:** `font-display text-[20px]`: "Meal cost configuration"

**Sub-heading explainer:** `font-sans text-sm text-text-secondary` ‚Äî "Edit these tables for accuracy. Changes recompute the protected floor immediately."

**Three editable tables, stacked vertically with `gap-sp-8` between:**

#### Table A: Meal definitions

Columns: Meal name | Type | Cost / Ingredients | Actions

- "Type" is a badge: "Decomposed" (success token) or "Flat cost" (text-secondary token)
- "Cost / Ingredients" cell: for decomposed meals, lists ingredient names (read from unit-cost map); for flat-cost meals, shows a `NumberInput` (dollar amount, `min-h-[44px]`)
- Stub rows (undefined meal from SMC) render with `text-warning` name and a "Needs ingredients" badge in warning colors
- Add row: `<PrimaryButton>` ‚Äî "Add meal definition"
- Delete row: `<DestructiveButton>` ‚Äî "Delete" (confirmation: see Copywriting Contract)
- Row height: `min-h-[44px]` for all action cells

#### Table B: Unit-cost map

Columns: Ingredient | Tag | Unit | Cost/unit | Actions

- "Tag" cell: `<SelectInput>` with options "Macro-bearing" / "Flavor / condiment"
- Unpriced rows (cost = 0 or null): row background `bg-warning bg-opacity-10 border-l-2 border-warning` ‚Äî visually flags the unpriced state (EDGE-03)
- "Cost/unit" cell: `<NumberInput>` inline, `min-h-[44px]`
- Add row: `<PrimaryButton>` ‚Äî "Add ingredient"
- Delete row: `<DestructiveButton>` ‚Äî "Delete"

#### Table C: Portion model

Columns: Ingredient | Portion | Unit | Actions

- "Portion" cell: `<NumberInput>` inline (grams or serving unit), `min-h-[44px]`
- Ingredients listed here are the union of all macro-bearing ingredients from the meal-definition table
- Add row: not available independently ‚Äî portions are added when an ingredient is added in Table B
- Delete row: `<DestructiveButton>` ‚Äî "Delete"

**FOOD-13 ‚Äî lastRefinedFromReceipts:**

Below Table B, render: `font-sans text-xs text-text-secondary` ‚Äî "Last refined from receipts: {date or 'Not yet recorded'}". Beside it, a `<SecondaryButton>` ‚Äî "Mark refined today". This records a timestamp, not a dollar edit. The ~$550 seed value is the starting point, convergence happens via Table B.

## Interaction States

### Floor line
- **Computed, clean:** value rendered, green badge "Plan current"
- **Computed, gaps:** value rendered (fallback-high applied), amber badge "Needs attention", expandable detail list
- **Stale / no plan:** value rendered (max(last-computed, high-water)), amber badge "Needs attention", staleness row in detail list
- **First-load (loading):** placeholder `‚Äî` in `text-text-disabled`, no badge

### Config tables
- **Row default:** `bg-surface-raised` row with sp-3 padding
- **Row with gap (unpriced/undefined):** `bg-warning bg-opacity-10 border-l-2 border-warning`
- **Row editing (inline):** border `border-accent` wraps the row (matches ExpensesPage EditRow pattern)
- **Row saving:** PrimaryButton shows disabled state; no double-submit
- **Delete confirmation:** inline confirmation row replaces action cell ‚Äî "Delete?" with DestructiveButton "Yes, delete" and SecondaryButton "Cancel"

### Accessibility
- Lock icon: `aria-hidden="true"` ‚Äî decorative
- Status badge/toggle: `aria-expanded={open}`, `aria-controls` pointing to detail list id
- Detail list: `role="list"`, each item `role="listitem"`
- All tables: `<table>` with `<thead>` / `<tbody>` / `<th scope="col">` for column headers; `<th scope="row">` for meal/ingredient name cells
- All interactive inputs: `min-h-[44px]` (touch target floor)
- Focus ring: `focus-visible:outline focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:outline-accent` (matches AppShell nav pattern)
- Color is never the sole differentiator: amber gap rows also carry a visible text flag, not just the border color
