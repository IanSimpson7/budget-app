## Surface 1: Income Dashboard (`/dashboard`)

### Layout
- Single-column, mobile-first. Full phone width with `px-sp-4` on mobile, `px-sp-6` on sm+.
- Section heading: "Income · {Month YYYY}" using `font-display text-[20px]`.
- Income bar: full-width, `h-[28px]`, `rounded-sm`, `border border-surface-border`.
- Three metric cards below the bar in a row: "Month to date", "Projected month", and either "Surplus" or the backfill alert card. On mobile, three cards in a single row each at `w-1/3`; if labels truncate, secondary label wraps below value.

### Income Bar Specification
```
[=============================|----]|
 ^actual fill (success)       ^proj ^bar end
                              ^defended-line tick (accent, 2px vertical)
       passive floor band edge (text.secondary label "floor")
```
- Bar track background: `surface.raised`
- Actual income fill (`success`, solid): width = `(totalIncomeMTD / projectedMonth) × 100%`
- Projected ghost segment (`warning` 40% opacity): extends from actual fill edge to `(projectedMonth / projectedMonth) × 100%` — only visible when `projectedMonth > totalIncomeMTD`
- Passive floor marker: vertical dashed line at `(passiveFloor / projectedMonth) × 100%` — `text.secondary`, 1px dashed, labeled "floor" in `text-xs text.secondary` below bar
- Defended-line tick: vertical solid line at `(defendedLine / projectedMonth) × 100%` — `accent.DEFAULT`, 2px solid, labeled "$3k" in `text-xs accent.DEFAULT` below bar
- When `projectedMonth > defendedLine × 1.2`: bar overflows gracefully — clamp bar track at 100%, display overflow amount in text below bar ("above defended line by $X")
- Bar is purely presentational — no interactive affordance on the bar itself

### Three Metric Cards
Each card: `bg-surface-raised border border-surface-border rounded-sm p-sp-3` with:
- Label: `font-sans text-xs text.secondary` (e.g. "Month to date")
- Value: `font-mono text-[20px] text.primary`
- Subtext (optional): `font-sans text-xs text.secondary`

**Surplus card (normal state):**
- Label: "Surplus"
- Value: `font-mono text-[20px] success` — e.g. "$0.00" or "$424.10"
- Subtext: none

**Backfill alert card (replaces surplus card when payroll-projection < defended line):**
- Card: `border-warning` (1px solid warning token) + `bg-warning/10` (10% opacity overlay via Tailwind arbitrary: `bg-[#9a7a2e]/10`)
- Label: `font-sans text-xs warning` — "Backfill alert"
- Value: `font-mono text-[20px] text.primary` — projected payroll amount, e.g. "$2,424"
- Subtext: `font-sans text-xs text.secondary` — "below $3,000 — add sessions to defend"
- No icon required — copy carries the signal

### Entry Link
Below the three cards: `SecondaryButton` "Add check" linking to `/entry`. Full-width on mobile.

### Empty State (no checks logged this month)
- Bar renders at 0% fill, ghost segment at 0 as well
- All three cards show `font-mono "$0.00"` with `text.disabled` color
- Below cards: `font-sans text-sm text.secondary` — "No income recorded for {Month YYYY}. Enter your first check."
- "Enter your first check" is a `<Link>` styled as `accent.DEFAULT underline` to `/entry`

## Surface 2: Check Entry (`/entry`)

### Layout
- Single-column, mobile-first. Max-width `max-w-[480px]` on desktop, centered.
- Section heading: "Add Income" — `font-display text-[20px]`
- Two modes toggled by a tab control: **"Manual entry"** (default) and **"Paste & parse"**
- Tab control: two buttons side by side, full-width. Active tab: `bg-surface-raised border-b-2 border-accent text.primary`; inactive: `bg-surface.DEFAULT text.secondary hover:text.primary`. `min-h-[44px]` on each tab.

### Manual Entry Tab
Form fields in a `bg-surface-raised border border-surface-border rounded-sm p-sp-4 sm:p-sp-6` card, `gap-sp-4` between fields:

| Field | Component | Type | Validation |
|-------|-----------|------|------------|
| Date | native `<input type="date">` styled to tokens | required | must be valid date; defaults to today |
| Net amount | `NumberInput` | required | must be > 0 |
| Source | `<input type="text">` styled to tokens | required | non-empty; shown with autocomplete from known-source list |
| Category | `<select>` styled to tokens | required | payroll / gift / other; defaults to payroll |
| Taxable | toggle / `<input type="checkbox">` | required | default ON for payroll, OFF for gift; matches category default |
| Note | `<textarea>` styled to tokens, 3 rows | optional | no validation |

All inputs: `bg-surface-raised border border-surface-border rounded-sm px-sp-3 py-sp-2 font-sans text-sm text.primary min-h-[44px] focus-visible:outline focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:outline-accent placeholder:text.disabled`

Financial amount: `font-mono` for the NumberInput.

Label pattern (from Phase 1 NumberInput): `font-sans text-xs font-semibold text.secondary` above input; helper text `font-sans text-xs text.secondary` below input; error text `font-sans text-xs destructive.DEFAULT` below helper.

**Surplus badge:** When the entered date is in a calendar month that already has 2 payroll checks, show an inline badge next to "Net amount" or below the date field:
- `bg-success/15 text-success font-sans text-xs rounded-sm px-sp-2 py-sp-1` — "3rd check this month — will be classified as surplus"
- This is informational, not blocking.

**Known-source autocomplete:** Source input shows a dropdown of previously confirmed source strings (from the known-source-list in settings). Dropdown items: `bg-surface-raised border border-surface-border rounded-sm` list, each item `min-h-[44px] px-sp-3 py-sp-2 font-sans text-sm text.primary hover:bg-surface-border cursor-pointer`. When a known source is selected, category and taxable auto-fill to the remembered values — shown as a brief non-dismissable inline note: `font-sans text-xs text.secondary` "Auto-filled from remembered source".

**Primary CTA:** `PrimaryButton` "Save check" — full-width on mobile, auto-width on desktop. Disabled when any required field invalid.

**Success state:** On save, `Toast` (success variant, Phase 1 component) — "Check saved." Auto-dismisses after 3s. Form resets to defaults (date = today, amount = empty, source = empty).

### Paste & Parse Tab
Two-step flow: **step 1 = paste input**, **step 2 = confirm table**.

**Step 1 — Paste input:**
- `<textarea>` full-width, `min-h-[160px]`, same token styling as other inputs but with `resize-y`
- Label: `font-sans text-xs font-semibold text.secondary` — "Paste transaction text"
- Helper: `font-sans text-xs text.secondary` — "Paste your checking statement block. Rows are parsed automatically."
- `PrimaryButton` "Parse entries" below the textarea. Disabled when textarea is empty.
- Parsing is synchronous client-side — no loading state required unless >200ms (unlikely).

**Step 2 — Confirm table (after parse):**
Replace the textarea + parse button with the confirm table. A `SecondaryButton` "Back — re-paste" top-left allows returning to step 1.

Confirm table container: `bg-surface-raised border border-surface-border rounded-sm overflow-x-auto`.

Table structure (each row is one parsed entry):
- Checkbox column: `<input type="checkbox">` styled, `min-h-[44px] w-[44px]` — default-checked for auto-income rows, unchecked for non-income (per D-05 logic)
- Date: `font-mono text-sm text.primary` — ISO date, read-only display (non-editable in table)
- Amount: `font-mono text-sm` — positive credits in `success`, negative debits in `text.secondary` — read-only
- Source: `font-sans text-sm text.primary` — inline editable `<input>` on row click/tap, `min-h-[44px]`
- Category: `<select>` payroll/gift/other, `min-h-[44px] font-sans text-sm` — pre-filled from known-source memory or parse signal
- Taxable: `<input type="checkbox">` — pre-filled to category default
- Note: truncated to 40 chars, `font-sans text-xs text.secondary`, non-editable (raw block text preserved)

Row states:
- Checked row: normal contrast (`text.primary`)
- Unchecked row: `text.disabled` on amount, source, category columns — visually dimmed but not hidden
- Row where auto-checked rationale applies (PAYROLL signal): no special styling; default-checked state alone communicates it
- Row that is a debit: unchecked by default, amount shown as `text.secondary` (no red — it is not an error)

Table header row: `font-sans text-xs font-semibold text.secondary uppercase tracking-wide` — "Date / Amount / Source / Category / Tax / Note"

Below the table: count summary `font-sans text-xs text.secondary` — "{N} of {total} entries selected".

**Primary CTA:** `PrimaryButton` "Commit {N} checks" — updates the label reactively with the checked count. Disabled when 0 entries checked.

**Commit result:** Toast "Saved {N} checks." + table replaced by a brief confirmation summary with a `SecondaryButton` "Add more" (returns to step 1) and a `SecondaryButton` "View dashboard" (navigates to `/dashboard`).

**Empty parse result** (no parseable rows found):
- Step 2 does not render. Remain on step 1 with an inline error below the textarea:
  `font-sans text-sm destructive.DEFAULT` — "No transaction rows found. Check the format and try again."
- Parse button re-enabled so Ian can retry.

## Interaction Contracts

### Reduced Motion
All transitions must respect `prefers-reduced-motion: reduce`. Phase 2 UI has:
- `transition-colors duration-150 ease-out` on buttons (existing pattern from PrimaryButton) — acceptable; color-only transitions are not motion per WCAG
- Income bar fill width: no animated transition on initial render. On new check commit, bar width updates instantaneously (no width animation). Reasoning: financial values must be readable, not theatrical.
- Toast entrance: existing Phase 1 Toast uses no motion — keep as-is.

### Focus Management
- After "Save check" in manual entry: focus returns to the Date field (first field) for fast re-entry.
- After "Commit N checks" in paste-parse: focus moves to "Add more" button.
- After "Back — re-paste": focus moves to the textarea.

### Keyboard Operability
- Tab order in manual entry form follows visual order: Date → Amount → Source → Category → Taxable → Note → Save.
- Confirm table checkboxes are reachable via Tab. Space toggles. Enter on a checked row expands inline edit for Source.
- Tab switcher (Manual / Paste & Parse): arrow keys move between tabs per ARIA tabs pattern.

### Form Validation Timing
- Validate on blur (not on every keystroke) for all text inputs.
- Validate on change for `NumberInput` (existing Phase 1 pattern from NumberInput component).
- Never show an error for an empty field the user has not yet touched.

## Accessibility Baseline

- Semantic HTML: `<form>`, `<label for="...">`, `<fieldset>` + `<legend>` for the tab group and category/taxable pair.
- All inputs have an associated `<label>`.
- Income bar: `role="meter"` with `aria-valuenow`, `aria-valuemin="0"`, `aria-valuemax` set to `projectedMonth`. `aria-label="Month-to-date income"`.
- Backfill alert card: `role="alert"` so screen readers announce it on dynamic appearance.
- Confirm table: `<table>` with `<thead>` / `<tbody>`, `<th scope="col">` headers.
- All icon-only buttons (if any): `aria-label`.
- Focus-visible ring: `focus-visible:outline focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:outline-accent` (established in PrimaryButton; apply consistently to all interactive elements).
- Contrast: `text.primary` (#e8dcc8) on `surface.DEFAULT` (#161210) = ~11:1 — passes WCAG AAA. `text.secondary` (#9a8e82) on `surface.DEFAULT` = ~4.6:1 — passes AA.
